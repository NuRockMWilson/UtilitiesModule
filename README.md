# NuRock Utilities AP

Automated accounts-payable processing for NuRock's utility bills. Receives bills from a shared inbox or drag-drop, extracts the billing fields via Claude vision, auto-codes them to the 500-XXX-XXXX.YY GL format, runs a trailing-baseline variance check, routes flagged bills to property managers for explanation, queues approved bills for Sage posting, and produces Excel trackers that match the legacy workbook format. Standalone Next.js 14 app today; designed to merge into the NuRock management dashboard.

---

## Prerequisites

- Node.js 20+
- A Supabase project (free tier is fine for dev)
- An Anthropic API key
- A Resend account and verified sending domain (for variance inquiry emails)
- Optional for production: Vercel account for deploy, a shared inbox that forwards attachments to the ingest webhook

## Local setup

```bash
git clone https://github.com/NuRockMWilson/nurock-utilities-ap.git
cd nurock-utilities-ap
npm install
cp .env.example .env.local
```

Edit `.env.local` and fill in at minimum:

- `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY` — from Supabase project settings
- `SUPABASE_SERVICE_ROLE_KEY` — server-only, keep out of version control
- `ANTHROPIC_API_KEY` — for bill extraction
- `NEXT_PUBLIC_APP_URL` — `http://localhost:3050` locally

The rest (Resend, Sage paths, intake webhook secret) can wait until you're ready for those flows.

### Supabase setup

```bash
# If you have the Supabase CLI
supabase link --project-ref <your-project-ref>
supabase db push
```

Or from the Supabase dashboard SQL editor, run the five files in order:

1. `supabase/migrations/0001_initial_schema.sql` — creates all core tables, enums, the `v_property_summary` view, and RLS policies
2. `supabase/seed/0002_seed_reference_data.sql` — inserts the 19 properties and 11 GL accounts from the Utilities procedures doc
3. `supabase/seed/0003_storage_setup.sql` — creates the private `invoices` and `sage-exports` storage buckets with policies
4. `supabase/migrations/0004_sage_batches.sql` — creates the `sage_batches` table that tracks each AP Import file's lifecycle
5. `supabase/seed/0005_historical_data.sql` — backfills 599 monthly summary rows (2026 YTD) and 1,949 water meter readings (2013–2026) across 17 properties. Also adds property 611 (Residences at Beverly Park) which wasn't in the original Utilities procedures doc. This file is regenerated by `scripts/import-historical.py` — see the Historical import section below.

### Run

**Easiest (Windows):** double-click `start.bat` in the project root. It checks prerequisites, installs dependencies on first run, starts the dev server in a separate window, waits for it to boot, and opens Chrome to http://localhost:3050. To stop, close the "NuRock Dev Server" window (or run `stop.bat`).

**Manual:**

```bash
npm run dev
```

Open http://localhost:3050 — you'll be redirected to the sign-in page. Enter your NuRock email, then enter the **6-digit code** that arrives by email. (The app uses OTP instead of magic links so it works cleanly on mobile without bouncing between email and browser apps.)

First time signing in? Because the sign-in flow uses `shouldCreateUser: false` to keep the app closed to random signups, you need to create your user in Supabase first. In your Supabase project → **Authentication → Users → Add user → Create new user**, enter your NuRock email, click **Create user**. Then come back to the app login and request a code.

Once signed in, `user_profiles` will be empty. Promote yourself to admin from the Supabase SQL editor:

```sql
insert into user_profiles (id, email, full_name, role, active)
values (auth.uid(), 'you@nurock.com', 'Your Name', 'admin', true);
```

(Or run it as a superuser with a hardcoded UUID — whatever's easier.)

### Branded OTP email (optional)

Supabase's default email shows a magic link AND the code. Since the app only uses the code, you can swap in a cleaner code-only template. In Supabase → **Authentication → Email Templates → Magic Link → Message Body (HTML)**, paste the contents of `supabase/email-templates/otp-sign-in.html`. While you're there, consider shortening **Email OTP Expiration** from 3600s (1 hour) to 600s (10 min) — tighter security, still plenty of time to receive and type the code.

## Architecture at a glance

```
bills land → extraction → coding → variance check → approval → Sage post → payment → Excel export
(inbox/upload)  (Claude)    (GL lookup)  (baseline mean)         (300 CRE/Intacct)
```

### Stack

- **Next.js 14** App Router, TypeScript, Tailwind (NuRock brand tokens baked in)
- **Supabase** Postgres + Auth + Storage + Row-Level Security
- **Anthropic Claude** for PDF extraction via the vision API
- **Resend** for transactional email (variance inquiries with branded HTML templates)
- **ExcelJS** for legacy-format Summary workbook export
- **Sage adapter** layer — file-based for 300 CRE today, REST stub for Intacct migration

### Key concepts

**Variance baseline.** Each utility account has its own `variance_threshold_pct` (default 3.00%) and `baseline_window_months` (default 12). When a bill arrives, we take the mean of prior `approved`/`posted_to_sage`/`paid` bills in that window, excluding any with `exclude_from_baseline = true` and excluding any that were flagged but never explained. We normalize by days (daily usage for metered utilities, daily dollars for flat-rate) so a 32-day billing period doesn't falsely spike against a 28-day one. If the current daily rate is more than 3% above the mean, `variance_flagged` goes true and the bill routes to `needs_variance_note` status — it can't be approved until either AP saves an explanation or the property manager replies to the inquiry email.

**GL coding.** `500-XXX-XXXX.YY` where 500 is fixed, XXX is the property code, XXXX is the GL account, YY is the sub-code (for house vs. clubhouse vs. vacant splits). Bills auto-code when their extracted `account_number` matches a row in `utility_accounts`. Unknown accounts route to `needs_coding` with a best-guess GL from vendor-label inference so the AP clerk only has to confirm.

**Sage adapter.** Each property's `sage_system` column decides whether the post-to-Sage action runs the 300 CRE file generator (pipe-delimited AP Import file stored in the `sage-exports` Supabase Storage bucket, handed to Sharon as a signed download URL, then Sharon imports it via AP Tasks → Import Invoices and clicks "Confirm Sage import" in the app) or the Intacct REST adapter (stubbed — fill in `src/lib/sage/intacct.ts` when migration is live; posts authoritatively via API, no confirmation step needed). Batches have their own lifecycle table (`sage_batches`) so Sharon can re-download a file weeks later or void a batch that Sage rejected.

**Audit log.** Every state transition writes to `approval_log` with the actor's email and timestamp. Every invoice detail page shows the full history. This is the digital replacement for the paper trail on the yellow folders.

## Integration setup

### Shared inbox (recommended for vendor e-bills)

1. Create or designate `ap-utilities@nurock.com`
2. Point a Resend, SendGrid, or Cloudflare Email Routing inbound rule at `${NEXT_PUBLIC_APP_URL}/api/ingest`
3. Set `INTAKE_WEBHOOK_SECRET` to a random string; the inbound service must set it as the `X-Intake-Secret` header
4. Vendors that support e-billing get their delivery address updated to that inbox

### Folder watcher (recommended for scanned mail + manual vendor portal pulls)

For bills arriving via mail (Dakota scans them) and manually-pulled e-bills, the companion watcher script in `watcher/` runs on a NuRock workstation that has the H: drive mapped. Drop a PDF into `H:\Claude\Utility Invoices` and within ~30 seconds it's in the Invoices inbox with extraction completed.

```
cd watcher
npm install
cp .env.example .env
# edit .env — APP_URL, INTAKE_WEBHOOK_SECRET (same value as web app), WATCH_FOLDER
npm start
```

Install as a Windows service for production with `npm install node-windows && npm run install-service` so it survives reboots and auto-restarts on crash. See `watcher/README.md` for full configuration and troubleshooting.

Processed files move to `H:\Claude\Utility Invoices\processed\YYYY-MM-DD\` automatically; failures move to `failed\YYYY-MM-DD\`. Content hashing means re-processing the same file is a no-op — the watcher notices the existing invoice and moves on.

### Sage 300 CRE

- No filesystem or LAN dependency — the whole flow runs on Vercel.
- When invoices are posted, the adapter generates the pipe-delimited AP Import file in memory, uploads it to the private `sage-exports` Supabase Storage bucket, and hands Sharon a 7-day signed download URL.
- Sharon opens the generated file, runs **AP Tasks → Import Invoices** in the Sage 300 CRE client, then clicks **Confirm Sage import** in the app. That confirmation step flips the invoices from `approved` to `posted_to_sage`.
- If something goes wrong in Sage (rejection, wrong account balance, etc.) she can **void** the batch from the batch detail page — invoices drop back to `approved` and can be regenerated into a fresh batch. Confirmed-posted batches cannot be voided; correct those in Sage with an offsetting entry.
- Single-invoice posting (from the invoice detail page) and multi-invoice batching (from the Payment queue) both route through the same `createBatch()` function and produce the same artifact format. Sharon will generally prefer the Payment queue workflow — one file per Wednesday's approved batch, not N separate files.
- If your Sage install uses a custom .apn import layout, adjust the column order in `src/lib/sage/sage300cre.ts`.

### Sage Intacct

- Fill in `SAGE_INTACCT_*` env vars once the migration is live
- Replace the stub in `src/lib/sage/intacct.ts` with real REST calls to `POST /services/v3/objects/accounts-payable/bill`
- Flip individual properties from 300 CRE to Intacct by updating `properties.sage_system`
- Intacct posting is authoritative via API — no artifact, no confirmation step; invoices flip straight to `posted_to_sage`.

### Resend (variance emails)

- Verify `nurock.com` as a sending domain
- Set `RESEND_API_KEY` and `VARIANCE_FROM_EMAIL`
- For reply-tracking, configure inbound routing for `variance-reply+*` to a webhook (build this in the next phase)

## Historical import

The `scripts/import-historical.py` Python script parses NuRock's legacy per-property Excel workbooks and multi-year water-usage breakdown files, emitting `supabase/seed/0005_historical_data.sql` that loads everything into the database. The seed file that ships with this repo was generated against the full set of 2026 workbooks and water breakdowns you provided, giving the variance engine real baselines from day one:

- **599 monthly Summary rows** across 17 properties (2026 YTD actuals by GL, feeding the tracker)
- **1,949 water meter readings** across 16 properties (2013–2026 — this is where multi-year water usage lives)
- Property 611 (Residences at Beverly Park) backfilled

### Regenerating the seed

If you add new source files or adjust the parsers, regenerate the seed:

```
pip install openpyxl                            # one-time
python3 scripts/import-historical.py \
    --input /path/to/historical/files \
    --output supabase/seed/0005_historical_data.sql
```

The script:
- Auto-detects whether each file is a per-property workbook (e.g. `555Sunset_Pointe.xls`) or a water usage breakdown (e.g. `Sunset_Pointe_Water_usage_break_down.xlsx`)
- Handles four Water-breakdown layouts observed across the files (Service Period, Service dates, Read Date/Consumption History, and CCF vs gallons)
- Tolerates the slight sheet-name variations across files (`Summary` vs `Summary ` with trailing space, `Water` vs `Water-Sewer` vs `Water & Sewer`)
- Is idempotent — re-running produces identical SQL, and the SQL itself uses `ON CONFLICT DO NOTHING` so applying twice is safe
- Requires LibreOffice on PATH for .xls → .xlsx conversion (the Mac/Linux default) — on Windows, pre-convert files with Excel's Save As first

### What the historical invoices look like

Summary rows land as `status='paid'` invoices with:
- `invoice_number` prefixed `HIST-S-` and a deterministic suffix (`HIST-S-555-5120-Water-2026-03`)
- `source_reference = 'historical_import_summary'`
- One row per property × GL × month × description

Water readings land as `status='paid'` invoices with:
- `invoice_number` prefixed `HIST-W-` (`HIST-W-555-20130119`)
- `source_reference = 'historical_import_water_usage'`
- Linked `usage_readings` rows with usage_amount, usage_unit (gallons / ccf), days, occupancy_pct

Both prefix patterns make it trivial to exclude historical rows from real AP workflows or delete them wholesale if needed: `DELETE FROM invoices WHERE source_reference LIKE 'historical_import_%'`.

## Deploy to Vercel

1. Import the repo in Vercel
2. Copy all env vars from `.env.local`, but use production Supabase keys
3. Set `NEXT_PUBLIC_APP_URL` to the Vercel domain
4. In Supabase Auth settings, add the Vercel domain to the redirect allowlist
5. Deploy

The `/api/ingest` and `/api/ingest/pdf` endpoints are excluded from auth middleware so email providers and the folder watcher can POST without a session cookie — they're protected by `INTAKE_WEBHOOK_SECRET` instead.

## Standalone → Dashboard

When the central dashboard is ready, this app can merge in one of two ways:

1. **Module merge** — Move `src/app/(app)/*` into `apps/utilities-ap/` inside the dashboard monorepo. The Supabase schema stays in its own schema (`utilities_ap`) inside the shared project. Lowest risk.
2. **Schema merge** — Migrate these tables into the dashboard's Supabase project, update RLS to integrate with the central user/role system, and redeploy. More work but fewer moving parts long-term.

Either way, no business-logic code needs to change — the adapters, variance engine, and coding resolver are framework-agnostic TypeScript.

## Conventions

- Dollar amounts: `$1,234,567.89` format via `formatDollars()`
- Percentages: exactly two decimals, e.g. `3.00%` via `formatPercent()`
- All writes go through `transitionInvoice()` or the equivalent so the audit log stays complete
- Sentence case in UI headings (no Title Case)
- NuRock brand tokens live in `tailwind.config.ts` — navy `#164576`, tan `#B4AE92`, Oswald for display, Inter for body
